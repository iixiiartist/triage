## 2026-04-23 — DECISION — MASSIVE integration scope + `out_of_scope` label

**Context.** The first eval-set freeze run produced 100% English test data because the MASSIVE loader was failing silently — `datasets` 4.0 dropped loader-script support (confirmed via `debug_massive_load.ipynb`), and the per-locale `try/except` in `load_massive` was swallowing the `RuntimeError: Dataset scripts are no longer supported`. Migrating to the `mteb/amazon_massive_intent` parquet mirror fixed the load, but surfaced a second problem: MASSIVE's ~60 intents are overwhelmingly voice-assistant commands (alarms, music, IoT, weather, email, etc.), and `INTENT_REMAP` only explicitly mapped 7 of them. The remaining 53 fell through to `other`, which became 62.5% of the test set and made SAM-Intent F1 structurally unreadable (trivial always-predict-`other` would score 62.5%).

**Decision.** (a) Introduced a new intent label `out_of_scope`, distinct from `other`. `other` = legitimate support query without a specific bucket; `out_of_scope` = not a support query at all. Downstream routing: `out_of_scope` → `auto_close`, because these queries shouldn't consume a human queue slot. (b) Every one of MASSIVE's 60 intents is explicitly mapped in `INTENT_REMAP` to `out_of_scope`; no fall-through allowed for MASSIVE labels. (c) MASSIVE is downsampled to `MASSIVE_SAMPLE_PER_LOCALE = 2000` per locale (~20k total vs. the full 115k), preserving 10-locale language diversity for SAM-Lang without drowning Bitext's support-intent signal.

**Data/Evidence.** All 60 MASSIVE intents classified manually against the rule "does a customer support team do this?" — 60/60 came out as voice-assistant commands. Predicted post-remap test distribution: `out_of_scope` ~48%, `other` ~6.4%, every Bitext support class clears the `MIN_TEST_PER_CLASS = 30` threshold. Alternatives considered: pinning `datasets<4.0` (rejected — reverses audit-aligned pin, parks on EOL track); collapsing everything into `other` (rejected — loses signal; "not a support query" and "support query without a bucket" are different NICO routing decisions); excluding MASSIVE from intent training entirely (rejected — adds a second data pipeline for marginal benefit).

**Implication.** (1) `spam` has no training data in v1 — `general_joke` was its only source and now maps to `out_of_scope` (not spam; spam implies adversarial, `general_joke` is legitimate-but-off-topic). `spam` will be demoted to `other` by low-support pruning. If needed in v2, source from Twitter or adversarial sets. (2) `out_of_scope` as a single dominant class (~48% of test) is an architectural characteristic, not a bug — the model is expected to learn "this is voice-assistant-style traffic, auto-close it" as a primary capability, not an edge case. Report SAM-Intent F1 both including and excluding `out_of_scope`, since the class is easy enough to identify that aggregate F1 would mask performance on the 13 real support intents. (3) Silent `try/except` in data loaders is now a known anti-pattern: the fixed `load_massive` prints exception class names prominently and reports loaded-N/M locale totals so a silent-all-fail can't happen twice.

**Next action.** Re-run `phase0_eval_set_freeze.ipynb` on Colab with the patched remap, produce a new `test_set_hash.txt`, commit the hash + `class_distribution.md` + `spot_check_samples.md` + the updated notebook. The intermediate hashes `b3921aea…` (en-only) and `0ea2d8cb…` (62%-`other`) are discarded — only the hash produced after the remap is approved is the keeper.
